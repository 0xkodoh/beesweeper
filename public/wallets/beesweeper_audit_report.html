🔴 Critical — security

No security headers in next.config.ts

The config file is completely blank. No CSP, no framing protection, no MIME-type enforcement.

CRITICAL
Add a headers() function to next.config.ts. At minimum:

X-Frame-Options: DENY — prevents clickjacking of the wallet connect flow.
X-Content-Type-Options: nosniff — stops MIME sniffing attacks.
Content-Security-Policy — whitelist only the domains your app talks to (Supabase, Base RPC, Reown).
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=()

Without CSP, a supply chain attack on any npm package can inject arbitrary scripts that steal connected wallet signatures.

SUPABASE_SERVICE_ROLE_KEY must be server-side only

Listed in .env.example. If any client component imports it, the key is bundled and publicly exposed.

CRITICAL
The service role key bypasses all Supabase Row Level Security. It must NEVER appear in NEXT_PUBLIC_* form and must never be imported in any file inside app/ that renders on the client.

Audit every file that uses it: it should only appear in app/api/ route handlers (server-side). If you need it in a Server Component, that's also fine — but confirm it's not in a 'use client' file.

Also verify Supabase RLS is enabled on your scores and leaderboard tables — don't rely solely on the anon key being read-only.
🟡 Medium — security & integrity

Client-side score submission — no server verification

Game logic runs entirely in the browser. Any score can be fabricated and submitted onchain.

MEDIUM
This is the fundamental trust problem with all client-side onchain games. Any player with DevTools can set their score to any value and submit it. Options ranked by effort:

Easy win: Add a server-signed score (Next.js API route signs the score + wallet + timestamp with your server private key; the smart contract verifies the signature via ecrecover). This is the most practical fix.

Harder: Move game logic to a server or ZK-proof the game state. Probably overkill for a Bee's Academy submission.

At minimum, disclose in your README that scores are self-reported and unverified.

Smart contract not independently auditable without Basescan verification

If the contract at 0xA130B42… isn't verified on Basescan, reviewers and users can't read its logic.

MEDIUM
Verify immediately if not done. The hardhat.config.ts file references BASESCAN_API_KEY in .env.example but does NOT include the etherscan config block needed for verification. Add this to your Hardhat config:

etherscan: { apiKey: { base: process.env.BASESCAN_API_KEY ?? "" } }

Then run: npx hardhat verify --network base 0xA130B42...

Without verification, Bee's portal reviewers and players have no way to audit whether the contract does what you say it does.

Contract-level concerns (unverifiable without source access)

Standard onchain game risks: no score bounds, no rate limiting, no pause mechanism.

MEDIUM
Without seeing the contract source I can't confirm, but common issues to check:

Score bounds: Does the contract enforce a max score? Without it, a player can submit type(uint256).max and permanently dominate the leaderboard.

Rate limiting: Can one address spam thousands of score submissions? Add a cooldown (e.g. 1 submission per address per 10 minutes).

Pause/admin: Can you pause submissions if something goes wrong? An Ownable + pause pattern gives you an emergency stop.

Events: Does the contract emit events on score submission? These are free to index and important for Supabase sync reliability.

RPC_URL hardcoded to Base Sepolia in .env.example

RPC_URL=https://sepolia.base.org is pre-filled. A contributor who copies this file will deploy to testnet.

MEDIUM
Rename the variable to BASE_SEPOLIA_RPC_URL and leave it blank in .env.example (just the key, no value), matching the pattern of BASE_MAINNET_RPC_URL=. Also add a deploy:base npm script to package.json alongside deploy:base-sepolia — the mainnet deployment script is missing.

For production, use a paid RPC (Alchemy, QuickNode) rather than the public Base endpoint — it has rate limits and no SLA.
🔵 Low severity

Reown AppKit project ID needs domain allowlisting

NEXT_PUBLIC_WALLETCONNECT_PROJECT_ID is exposed in the client bundle (expected), but needs dashboard protection.

LOW
Log into your Reown (formerly WalletConnect) Cloud dashboard and add beesweeper.vercel.app to the allowed domains for your project ID. This prevents other sites from abusing your quota with your project ID. Without it, your project ID can be scraped from the JS bundle and used by anyone.

PRIVATE_KEY in .env.example — check git history

The deployment wallet key is listed in .env.example. Ensure it was never committed with a real value.

LOW
Run git log --all -p | grep PRIVATE_KEY to confirm no real private key was ever committed. If one was, rotate the wallet immediately — even deleted commits can be recovered.

The PRIVATE_KEY in .env.example is only for Hardhat deployment. Confirm it is never referenced in any app/ directory file.

Bundle bloat: ethers.js + viem both included

wagmi v3 uses viem internally. Having ethers adds ~500KB to the client bundle unnecessarily.

LOW
Check if ethers is actually imported anywhere in app/. If not, remove it from package.json. wagmi v3 + viem covers everything you need for reading/writing contracts and formatting values.

Similarly, @walletconnect/ethereum-provider is already a transitive dependency of Reown AppKit — having it as a direct dependency risks version conflicts. Remove it unless you're using it directly.

lucide-react version "^1.14.0" doesn't exist

The published lucide-react library is in the 0.x range. Version 1.14.0 is not a real release.

LOW
Change to "lucide-react": "^0.383.0" or whatever the latest stable is. An invalid version won't resolve correctly and could silently install an unexpected version via semver fallback, or fail your build entirely if the constraint can't be satisfied.

Suspicious "accounts" dependency

accounts: "^0.8.10" is an unusual npm package. Likely unused or a mistake.

LOW
The accounts npm package is a minimal Ethereum/Bitcoin account utility. Grep your codebase for any import ... from 'accounts'. If nothing imports it, remove it — every unnecessary dependency is a supply-chain risk vector. This is especially important in a Web3 app where a compromised package could intercept wallet operations.
⬜ Cleanup / refactor

README is still the default Next.js boilerplate

The current README says "Getting Started with create-next-app" — zero BeeSweeper-specific content.

INFO
Before Bee's portal submission, replace the README with: game description, contract address + Basescan link, environment variables list with explanations, how to run locally, how score submission works, tech stack, and a screenshot or GIF of gameplay. This is public-facing and is what reviewers see first.

AGENTS.md / CLAUDE.md are AI dev artifacts in a public repo

These files describe your internal AI-assisted development workflow. Fine to keep, but worth noting.

INFO
CLAUDE.md just references AGENTS.md, which warns AI agents about breaking API changes in Next.js 16. These are harmless — no secrets — but they give away that the project was AI-generated, which reviewers might notice. Move them to a .dev/ folder or add them to .gitignore before final submission if you prefer a cleaner public profile.

No deploy:base (mainnet) npm script

package.json has deploy:base-sepolia but no equivalent for mainnet.

INFO
Add "deploy:base": "hardhat run scripts/deploy.ts --network base" to your scripts. The contract is already deployed, but if you ever need to redeploy or deploy a new version, the command isn't documented. Also add a verify:base script for convenience.

Only 4 git commits, no branching strategy

All code was committed in a handful of squashed pushes. No PR history or branching.

INFO
This is fine for an MVP, but for a public project submitted to a program portal, a visible git history with meaningful commits signals quality to reviewers. Going forward: use feature branches, write descriptive commit messages (feat: add hard mode leaderboard, fix: rate-limit score submissions), and keep main as your stable branch. It also makes it easier to roll back if something breaks.
Overall verdict: The project is solid for a Bee's Academy submission — the onchain score flow, Supabase leaderboard, and wallet integration all work. The two critical items (security headers + service role key isolation) should be fixed before publishing. The contract verification gap is the most important for reviewer trust. The score integrity limitation is inherent to client-side games — just disclose it.
